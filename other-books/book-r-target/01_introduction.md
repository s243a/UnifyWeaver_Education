# Chapter 1: Introduction to the R Target

## Why R for Prolog?

R is a language built around vectors, data frames, and functional programming. These
features make it a surprisingly good compilation target for Prolog predicates,
especially when the output will feed into data analysis or statistical workflows.

| Prolog | R |
|--------|---|
| Facts + Rules | Lists + Functions |
| Clause selection | `if`/`else if` chains |
| Backtracking | `sapply()` / `lapply()` iteration |
| Unification | Pattern matching via `==` / `switch` |
| Tail recursion | `for`/`while` loops with accumulator |
| `assert`/`retract` | `assign()` / environment mutation |
| Memoization | `new.env(hash=TRUE)` hash tables |

## Architecture

```
┌───────────────────────────────────────┐
│        UnifyWeaver (Prolog)           │
│  r_target.pl + r_bindings.pl          │
└─────────────────┬─────────────────────┘
                  │
                  ▼
           Generated .R script
                  │
                  ▼
         Rscript output.R <args>
                  │
                  ▼
           stdout / data frames
```

The R target lives in two files:

- **`r_target.pl`** (~1,400 lines) — Code generation for facts, rules, recursion patterns,
  pipelines, and data frame operations
- **`r_bindings.pl`** (~200 lines) — Maps 70+ Prolog built-ins to R equivalents

## Compilation Modes

The R target supports several compilation modes, selected by predicates and options:

| Mode | Entry Point | Use Case |
|------|-------------|----------|
| **Facts** | `compile_facts_to_r/3` | Prolog facts → R list data with accessor functions |
| **Predicate** | `compile_predicate_to_r/3` | Rules → R functions with if/else dispatch |
| **Pipeline** | `compile_r_pipeline/3` | Data frame operations with fixpoint iteration |
| **Recursion** | via `advanced_recursive_compiler` | Auto-detected: tail, linear, tree, mutual, multi-call |

```prolog
% Facts mode
?- compile_facts_to_r(parent, 2, Code).

% Predicate mode
?- compile_predicate_to_r(greet/1, [], Code).

% Recursion (auto-detected via pipeline)
?- compile_advanced_recursive(factorial/2, [target(r)], Code).
```

## Generated Script Structure

Every generated R script follows this structure:

```r
#!/usr/bin/env Rscript
# <predicate> - <pattern description> (R)
# Generated by UnifyWeaver

# Memoization setup (if needed)
pred_memo <- new.env(hash=TRUE, parent=emptyenv())

# Main function(s)
pred <- function(input, expected=NULL) {
    # Check memo
    # Base cases
    # Recursive/computed case
    # Memoize and return
}

# CLI entry point
if (!interactive()) {
    args <- commandArgs(TRUE)
    if (length(args) >= 1) cat(pred(as.integer(args[1])), "\n")
}
```

The `if (!interactive())` block makes scripts runnable from the command line while
still allowing `source()` in interactive R sessions.

## Running Generated Code

```bash
# Direct execution
Rscript output/advanced/factorial.R 6
# → 720

# With multiple arguments (comma-separated lists)
Rscript output/advanced/sum_list.R 1,2,3,4,5
# → 15

# From an interactive R session
source("output/advanced/factorial.R")
factorial(6)
# [1] 720
```

## Memoization Strategy

R doesn't have built-in memoization, but its environment objects provide O(1)
hash table lookups. UnifyWeaver generates memoization using this pattern:

```r
# Create a hash-table environment
pred_memo <- new.env(hash=TRUE, parent=emptyenv())

# Store: O(1) insert
pred_memo[["key"]] <<- value

# Lookup: O(1) check + retrieve
if (!is.null(pred_memo[["key"]])) {
    cached <- pred_memo[["key"]]
}
```

The `<<-` operator writes to the parent environment scope, allowing the memoization
table to persist across recursive calls. Using `parent=emptyenv()` prevents accidental
inheritance from the global environment.

## Multifile Dispatch

The R target registers itself with core recursion modules using Prolog's multifile
mechanism:

```prolog
% In r_target.pl:
:- multifile tail_recursion:compile_tail_pattern/9.
:- multifile linear_recursion:compile_linear_pattern/8.
:- multifile tree_recursion:compile_tree_pattern/6.
:- multifile multicall_linear_recursion:compile_multicall_pattern/6.
:- multifile direct_multi_call_recursion:compile_direct_multicall_pattern/5.
:- multifile mutual_recursion:compile_mutual_pattern/5.
```

When the advanced recursive compiler detects a pattern and the target is `r`, it
automatically dispatches to the R-specific code generator. No modifications to
core modules are needed to add or update R code generation.
